Viva Engage / Yammer - API-rapport & lärdomar

Hur ett helt Viva Engage-nätverk (Svenska kyrkan) skrapades och arkiverades till ett sökbart SQLite-arkiv inför nedstängning - vilka API:er som användes, vad som fungerade, och fallgroparna. Tänkt som referens för den som vill göra något liknande mot Viva Engage / Yammer.

Bakgrund & syfte

Viva Engage (f.d. Yammer) saknar en bra exportväg för historik. Microsoft Graph exporterar inte meddelandehistorik per community, så lösningen blev att kombinera två API:er: det gamla, stabila Yammer REST-API:t (full tråd-/meddelandestruktur) och det moderna Viva GraphQL-API:t (reaktioner, seen-count, medlemmar, storylines - sådant REST saknar). Allt drevs av en fångad webbsessions-token, inte en Entra-app.

Resultat i arkivet: 128 communities, ~30 300 meddelanden, 34 500 användare, 17 000 reaktionsrader (37 900 reaktorer), 296 storyline-trådar, 7 600 bilagor.

Sammanfattning (det viktigaste)

  • Två API:er kompletterar varandra. REST för innehållet, GraphQL för det REST inte ger.
  • Fångad sessionstoken (kortlivad, ~en timme) fungerar mot båda API:erna - även mot engage.cloud.microsoft/graphql, trots att tokenens audience är yammer.com.
  • Paginera på sid-fullhet, inte på "finns mer"-flaggor. Yammers older_available och GraphQL:s hasNextPage ljuger ibland.
  • GraphQL kör Apollo persisted queries - du skickar bara operationName + sha256Hash + variabler, aldrig query-texten. Hasharna dör vid app-deploy.
  • Förhandsdata är inte full data. featuredReactions ger max 8 reagerande; hela listan kräver ett eget paginerat anrop.

Arkitektur

Allt byggdes som idempotenta, resumebara pass som skriver rå-JSON till disk, följt av ett build-steg som dedupar till SQLite. Inget pass håller state i minnet som inte också finns på disk - dör token mitt i tappas inget arbete.

dump        in_group        -> data/raw/groups/<id>/page_*.json   (alla communities)
threads     in_thread       -> data/raw/threads/<id>/page_*.json  (backfilla hela trådar)
build       rå-JSON         -> data/archive.db (SQLite + FTS5, dedup på message-id)
enrich      GraphQL         -> reaktioner + seen-count per tråd
reactors    GraphQL         -> FULLA reaktörslistor (där urvalet räckte inte)
storylines  GraphQL         -> org-storyline-flödet -> backfill via in_thread
community   GraphQL         -> info-text, fästa länkar, medlemmar (+admin)
users       REST users.json -> fulla profiler + avatarer
download    REST            -> bilagor + länkkorts-miniatyrer

En FastAPI-kontrollpanel orkestrerar passen (start/stopp, progress, token-inklistring) och en separat läs-app serverar arkivet. En pipeline kör alla pass i serie.

Auth & token

Token fångas från den inloggade webbsessionen (ett userscript hookar sidans fetch och matar panelen). Den är kortlivad, så pass läser token färskt per anrop och väntar in en ny vid 401 i stället för att krascha ("self-heal").

Authorization: Bearer <token>
# Samma token mot båda baser:
#   https://www.yammer.com/api/v1/...
#   https://engage.cloud.microsoft/graphql
Insikt: tokenens aud är www.yammer.com, men den accepteras ändå av den moderna GraphQL-endpointen. Man behöver alltså inte en separat Entra-app - det räcker med det den inloggade användaren själv ser (delegerad åtkomst).

Legacy Yammer REST-API

Bas: https://www.yammer.com/api/v1. Grovt ~10 req/10s, så throttla (~1,2s mellan anrop) och backa av vid 429/5xx. Ger full tråd-/meddelandestruktur.

    { "id": 3791713089953792, "thread_id": 3791713089953792, "sender_id": 1471,
      "replied_to_id": null, "created_at": "2023/09/12 07:55:01 +0000",
      "body": { "plain": "...", "rich": "...", "parsed": "..." },
      "message_type": "update", "liked_by": { "count": 30, "names": [...] },
      "attachments": [...], "notified_user_ids": [...] } ],
  "references": [ { "type":"user","id":1471,"full_name":"...","mugshot_url":"..." } ],

Fallgrop: meta.older_available är opålitlig - den kan vara false trots att äldre meddelanden finns. Litar man på den trunkeras trådar (man tappar de äldsta meddelandena, inklusive trådstarten). Lösning: paginera på sid-fullhet - full sida (len == limit) => hämta äldre via older_than = min(id); ofullständig sida => klart. Med no-progress-skydd.
Bra att veta: users.json returnerar samma fält som users/{id}.json (ort, avdelning, bio, expertis, telefon, mugshot). Ett enda paginerat svep ger alltså hela nätverkets fulla profiler - ingen per-användare-loop behövs.

Modern Viva GraphQL (Apollo persisted queries)

Bas: POST https://engage.cloud.microsoft/graphql?operationName=X. Klienten kör APQ: query-texten skickas aldrig, bara ett operationName + en sha256Hash som identifierar den registrerade queryn, plus variabler.

POST https://engage.cloud.microsoft/graphql?operationName=MessageReactionsClients
Authorization: Bearer <token>
Content-Type: application/json
{
  "operationName": "MessageReactionsClients",
  "variables": { "messageId": "<base64>", "reaction": null, "after": null },
  "extensions": { "persistedQuery": { "version": 1,
    "sha256Hash": "5263008b3d71ffe625c7037ed657bc41457ab909ebb00a7c4798fada4048731c" } }
}

Nod-id är base64-kodad JSON

Alla GraphQL-id är base64 av {"_type":TYP,"id":"N"}:

gid("Message", 3791713089953792)  # base64 av {"_type":"Message","id":"3791713089953792"}
gid("Thread", n) / gid("Group", n) / gid("User", n)
# avkoda baklänges för att få heltals-id:t tillbaka

Så hittar man en hash

Hasharna finns inte dokumenterade - de fångas från den körande webbklienten: öppna funktionen i UI:t (t.ex. "vilka reagerade"), instrumentera sidans fetch, och läs operationName + extensions.persistedQuery.sha256Hash + variables ur anropet. (Nätverksfliken kan missa anropen beroende på verktyg - hooka fetch direkt.)

Fallgrop: hasharna byts vid app-deploy. Får man PersistedQueryNotFound är den utslängd ur cachen - den webbklient som är öppen återregistrerar den löpande, så behandla felet som transient (vänta/retry med en Viva-flik öppen). Dör den helt måste en ny hash fångas. Kör därför GraphQL-passen nära fångsten.

Fallgropar & insikter

1. "Finns mer"-flaggor ljuger - paginera på resultat

Både REST (older_available) och GraphQL (hasNextPage) kan säga "klart" för tidigt. Paginera på att en sida faktiskt är full / att nya poster kommer, inte på flaggan.

2. Cursor-riktningen är inte självklar

Storyline-flödena paginerar bakåt via startCursor + hasPreviousPage (inte endCursor/hasNextPage). Med fel fält fick vi nästan bara överlapp (samma 20 poster om och om). Rätt fält => rena, icke-överlappande sidor, hela flödet på ~18s.

Subtilt: samma flöde gick inte att paginera headless med olderThan: null + endCursor - men i webbläsaren fungerade det, eftersom UI:t sätter en riktig cursor vid scroll. Lösningen var startCursor, inte att efterlikna UI:t.

3. Förhandsdata != full data

featuredReactions ger bara ett urval på max 8 reagerande per typ (som Vivas UI:s "och N andra"). Hela listan kräver MessageReactionsClients per meddelande. Vi hämtade bara full lista där urvalet inte räckte (~700 av 17 000 grupper).

4. Signerade media-URL:er går ut

Mugshot- och länkkorts-miniatyr-URL:er är signerade och tidsbegränsade (?...&sig=). Sparar man dem och laddar ner senare => 403. Lösning: ladda via den stabila redirect-URL:en (302:ar till en färsk signerad asset) och spara filen lokalt - en extern CDN-länk dör ändå när tjänsten stängs.

5. Föräldralösa processer & zombies

Detacherade pass som överlever sin förälder fortsätter köra (och kan skriva över nyare data). Och en avslutad delprocess som inte reapats blir en zombie som os.kill(pid,0) rapporterar som "lever" - en panel kan fastna på "kör". Reapa barn (waitpid) och kontrollera /proc/<pid>/stat-tillståndet.

Datamodell (SQLite + FTS5)

Rå-JSON är sanningskällan; build dedupar till SQLite. Fritextsök via en FTS5-vy över body_plain. Storyline-poster lagras som vanliga meddelanden med group_id = NULL.

communities(id, full_name, privacy, company_group, moderated, restricted_posting,
            accessible, message_count, extended_description, member_count)
users(id, full_name, email, job_title, location, department, summary, ...,
      avatar_local, raw_json)
messages(id, group_id, thread_id, replied_to_id, sender_id, created_at,
         body_plain, body_rich, message_type, like_count, shared_message_id, ...)
attachments(id, message_id, type, name, web_url, local_path, description, thumb_local)
reactions(message_id, type, count)        reactors(message_id, type, user_id)
thread_meta(thread_id, seen_by_count)     mentions / likes / polls / pinned / group_members / tags
messages_fts USING fts5(body_plain, content='messages', content_rowid='id')

Mönster & rekommenderat upplägg

  • Rå-dump + separat build. Spara råsvaren oförändrade; bygg arkivet idempotent därifrån. Då kan du bygga om utan att skrapa om, och en schemaändring är bara en ny build.
  • Resumebara, idempotenta pass. Markörer på disk (.done/.skipped/ cursor-state). Naturlig resume: t.ex. "kapad reaktion" = sparade reaktorer < count -> passet hoppar automatiskt det som redan är komplett.
  • Self-heal-token. Läs token per anrop; vänta in en ny vid 401 i stället för att krascha.
  • Behandla persisted-query-fel som transient. Retry med en öppen webbklient som återregistrerar hashen; ge upp först efter en streak.
  • Separera styrning från visning. En admin-app (skrapning/token) och en läs-app (arkivet) som delar samma router men körs på olika portar - bara läs-appen exponeras externt.
  • Rendera säkert. Sanera body.rich via en allowlist (behåll formatering, skriv om @mention-spans till interna länkar) i stället för att lita på parsed.
  • Lazy-loading med fragment-endpoints för långa listor; spegla position i URL-hashen för delbara/återkomliga vyer.
